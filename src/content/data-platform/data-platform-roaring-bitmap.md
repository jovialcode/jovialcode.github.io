### Intro
`Procella`의 역인덱스를 설명하면서 `Roaring Bitmap`이라는 자료구조가 등장했다.
`YouTube`의 A/B 실험 쿼리 성능을 500배 끌어올린 주역으로 잠깐 언급하고 넘어갔는데, 이 자료구조 하나가 어떻게 그 수준의 성능 향상을 만들어냈는지 궁금하지 않을 수 없다.
#
`Roaring Bitmap`은 32비트 정수 집합을 압축해서 저장하고, 집합 간 교집합·합집합 같은 연산을 빠르게 수행하기 위해 설계된 비트맵 자료구조다.
`Druid`, `Elasticsearch`, `Apache Lucene`, `Apache Spark`, `ClickHouse` 같은 현대 데이터 시스템 대부분이 내부적으로 이 구조를 사용하고 있다.
#
이 구조를 제대로 이해하려면 먼저 비트맵이 왜 필요한지, 기존 비트맵의 어떤 한계가 있었는지부터 살펴봐야 한다.

### 비트맵과 역인덱스
검색 엔진이나 분석 시스템의 핵심 문제는 "이 조건을 만족하는 문서(또는 레코드)의 집합을 빠르게 찾는 것"이다.
예를 들어 `category = 'music'` AND `country = 'KR'` 조건을 만족하는 영상 ID 목록을 구해야 한다고 하자.

역인덱스(Inverted Index)는 이 문제를 이렇게 해결한다.

```
category=music → { 1, 4, 7, 10, 11, 15, ... }
country=KR     → { 2, 4, 7, 9,  11, 14, ... }
```

두 집합의 교집합 `{ 4, 7, 11, ... }`을 구하면 원하는 결과를 얻는다.

#

이 집합을 **비트맵(Bitmap)**으로 표현하면 교집합 연산이 매우 빠르다.
ID가 0부터 N까지 있을 때, N+1개 비트를 할당하고 해당 ID가 집합에 있으면 1, 없으면 0으로 표시한다.

```
category=music  0001001001010001
country=KR      0100101001010010

AND(교집합)     0000001001010000
```

AND 연산은 CPU 하나의 명령으로 64비트씩 처리할 수 있어 매우 빠르다.
#
문제는 **희소성(Sparsity)**이다.
100억 개의 영상 ID 중 100만 개만 `category=music`에 해당한다면, 100억 비트 중 100만 개만 1이고 나머지 9,990,000,000개는 0이다.
이 비트맵을 그대로 저장하면 10억 바이트(약 1.2GB)가 필요하다.
희소한 데이터에 메모리를 낭비하는 것은 물론, CPU 캐시 효율도 떨어진다.
#
이를 해결하기 위해 **WAH(Word-Aligned Hybrid)**, **EWAH**, **Concise** 같은 압축 비트맵이 등장했다.
연속된 0 또는 1 블록을 하나의 단어로 표현하는 RLE(Run-Length Encoding) 방식이다.
하지만 이 방식들은 공통적인 단점이 있다. **임의 접근(Random Access)이 불가능**하다는 것이다.
"1,234,567번 위치의 값이 무엇인가?"를 확인하려면 처음부터 순서대로 디코딩해야 한다.
#
`Roaring Bitmap`은 이 문제를 완전히 다른 접근으로 해결한다.

### Roaring Bitmap 핵심 아이디어
`Roaring Bitmap`의 핵심은 32비트 정수를 **상위 16비트**와 **하위 16비트**로 나누는 것이다.

```
┌────────────────────────────────────────┐
│         32비트 정수 (예: 131,175)      │
├──────────────────┬─────────────────────┤
│  상위 16비트 = 2 │  하위 16비트 = 103  │
│  (청크 번호)     │  (청크 내 위치)     │
└──────────────────┴─────────────────────┘

131,175 = 2 × 65,536 + 103
```

상위 16비트는 **청크 번호**로 쓴다. 청크는 최대 65,536개(0번부터 65,535번)이며, 각 청크는 하위 16비트 범위인 0부터 65,535까지의 값을 담을 수 있다.
하위 16비트는 **청크 내 위치**를 나타낸다.
#
구체적인 예를 보자. 집합 `{1, 2, 3, 100, 131,072, 131,175}`를 저장한다고 하면

| 값          | 상위 16비트 (청크) | 하위 16비트 (위치) |
|------------|------------------|------------------|
| 1          | 0                | 1                |
| 2          | 0                | 2                |
| 3          | 0                | 3                |
| 100        | 0                | 100              |
| 131,072    | 2                | 0                |
| 131,175    | 2                | 103              |

청크 0에는 값 `{1, 2, 3, 100}`이, 청크 2에는 값 `{0, 103}`이 들어간다.
청크 1은 비어있으므로 공간을 차지하지 않는다.

```
청크 인덱스 (정렬된 배열):
┌─────┬─────────────────────────────────┐
│ key │ container                       │
├─────┼─────────────────────────────────┤
│  0  │ → [1, 2, 3, 100]               │
│  2  │ → [0, 103]                     │
└─────┴─────────────────────────────────┘
```

이 청크 인덱스는 정렬된 배열이라 이진 탐색으로 O(log n) 접근이 가능하다.
WAH 같은 방식과 달리 임의 접근이 된다.
#
각 청크(컨테이너)는 담긴 값의 특성에 따라 세 가지 다른 방식으로 저장된다.

### 세 가지 컨테이너
**Array Container**는 하위 16비트 값들을 정렬된 `uint16` 배열로 저장한다.
값이 4,096개 이하일 때 사용하며, 메모리는 `값의 개수 × 2바이트`다.

```
Array Container (값 4개일 때)
┌────┬────┬────┬────┐
│  1 │  2 │  3 │100 │  → 4 × 2 = 8 바이트
└────┴────┴────┴────┘
```

값을 찾을 때는 이진 탐색을 사용하므로 O(log n)이다.
#
**Bitmap Container**는 65,536비트(8,192바이트 = 8KB)의 고정 크기 비트셋이다.
값이 4,096개를 초과할 때 사용하며, 메모리는 **항상 8KB로 고정**이다.

```
Bitmap Container (65,536비트 고정)
bit[0] bit[1] bit[2] bit[3] ... bit[65535]
  0      1      1      1    ...    0
         ↑      ↑      ↑
      해당 위치 값 존재
```

임의 위치 접근이 O(1)이며, AND/OR 연산은 CPU 비트 연산으로 초고속 처리된다.
#
**Run Container**는 연속된 값의 범위를 `(시작값, 길이-1)` 쌍으로 저장한다.
예를 들어 `{1, 2, 3, 4, 5}`는 `(1, 4)` 하나로 표현되고, `{1, 2, 3, 10, 11, 12}`는 `(1, 2), (10, 2)` 두 쌍으로 표현된다.

```
Run Container
{1, 2, 3, 10, 11, 12} → [(1, 2), (10, 2)]
                              ↑       ↑
                         시작=1    시작=10
                         길이-1=2  길이-1=2
                         (1–3)     (10–12)
```

타임스탬프처럼 연속적인 값이 많을 때 매우 효과적이다. `(start, length-1)` 쌍 하나가 4바이트이므로, 연속 범위가 많을수록 메모리가 크게 줄어든다.
#
컨테이너 타입은 **자동으로 결정되고 자동으로 전환**된다.

| 컨테이너       | 크기                     | 적합한 데이터        | 접근 복잡도 |
|--------------|--------------------------|---------------------|------------|
| Array        | 값 개수 × 2바이트 (최대 8KB) | 값이 적을 때 (≤4096) | O(log n)   |
| Bitmap       | 항상 8KB                  | 값이 많을 때 (>4096) | O(1)       |
| Run          | 범위 개수 × 4바이트        | 연속 범위가 많을 때   | O(log n)   |

#
**임계값이 4,096인 이유**는 수학적으로 명확하다.

```
Array Container(4096개)  = 4096 × 2바이트  = 8,192바이트
Bitmap Container         = 65536 / 8바이트 = 8,192바이트
```

값이 4,096개일 때 둘의 크기가 정확히 같다. 그 이상이 되면 Bitmap이 더 유리하므로, 4,096을 기준으로 타입을 전환한다.
#
Run Container를 쓸지 여부는 별도로 판단한다. 직렬화 시점에 "Run Container로 변환했을 때 크기가 줄어드는가?"를 확인하고 더 작은 쪽을 선택한다.
예를 들어 `{0, 1, 2, ..., 65534}`처럼 거의 모든 값이 있는 경우, Bitmap은 8KB지만 Run Container는 `(0, 65534)` 하나만 저장하면 된다.

### 집합 연산
`Roaring Bitmap`의 강점은 집합 연산에서 두드러진다.
두 비트맵을 AND/OR할 때, 컨테이너 타입 조합에 따라 최적의 알고리즘을 자동으로 선택한다.

| 연산 | Array vs Array | Array vs Bitmap | Bitmap vs Bitmap |
|-----|---------------|----------------|-----------------|
| AND | 작은 배열 순회 이진탐색 | 배열 원소를 비트맵에서 조회 | 64비트 AND 연산 |
| OR  | 정렬 병합 | 배열 원소를 비트맵에 삽입 | 64비트 OR 연산 |

#
가장 빠른 경우는 **Bitmap vs Bitmap**이다.
65,536비트를 64비트 단위로 나누면 1,024개의 `long` 값이다.
두 Bitmap Container를 AND하면 딱 1,024번의 비트 연산으로 끝난다.
현대 CPU는 SIMD 명령어로 이를 더 빠르게 처리할 수도 있다.
#
**Array vs Array** 교집합은 두 정렬된 배열을 갤로핑 탐색(Galloping Search)으로 병합한다.
크기 차이가 클수록 작은 배열의 각 원소를 큰 배열에서 이진 탐색으로 찾는 방식이 더 효율적이다.
`Roaring Bitmap`은 두 배열의 크기 비율을 보고 선형 병합과 갤로핑 탐색 중 더 유리한 방법을 동적으로 선택한다.
#
연산 결과 컨테이너의 밀도가 바뀌면, 예를 들어 OR 연산 후 값이 4,096개를 넘으면 Array에서 Bitmap으로 자동 변환된다.

### 실제 적용 사례
`Apache Druid`는 각 컬럼의 역인덱스를 `Roaring Bitmap`으로 저장한다.
`SELECT count(*) WHERE platform = 'mobile' AND region = 'KR'` 같은 쿼리가 들어오면 두 비트맵의 교집합을 구하고 cardinality를 반환한다.
`Roaring Bitmap` 도입 후 필터 쿼리 성능이 수십 배 향상되었다는 결과가 공식 문서에 기록되어 있다.
#
`Apache Lucene`(Elasticsearch의 코어)에서도 텀 인덱스를 `Roaring Bitmap` 기반으로 구현했다.
인덱스 세그먼트 병합 시 두 비트맵을 OR 연산으로 합치는 방식이 RLE 기반 구현보다 훨씬 빠르다.
`Apache Spark`의 집계 함수와 `ClickHouse`의 bitmap 내장 함수에도 `Roaring Bitmap`이 탑재되어 있다.
#
`Procella`에서는 `Artus` 파일 포맷의 역인덱스에 `Roaring Bitmap`을 사용한다.
A/B 실험 분석 쿼리는 특정 실험 그룹에 속한 사용자들의 데이터만 집계해야 한다.
"실험 그룹 A에 속한 사용자" 집합을 `Roaring Bitmap`으로 저장해두면, 수백만 행 중 해당 행만 빠르게 필터링할 수 있다.
논문에서는 이 최적화만으로 특정 쿼리의 실행 시간이 500배 단축되었다고 밝혔다.

### Outro
`Roaring Bitmap`이 특별한 이유는 "하나의 알고리즘으로 모든 데이터 분포를 잘 처리하겠다"는 욕심을 버린 데 있다.
희소한 경우에는 Array, 조밀한 경우에는 Bitmap, 연속적인 경우에는 Run Container를 선택하고, 데이터가 바뀌면 컨테이너를 자동으로 전환한다.
단순하지만 실용적인 이 설계가 수십억 행을 다루는 시스템에서도 예측 가능한 성능을 보장한다.
#
오늘날 대부분의 오픈소스 분석 엔진이 `Roaring Bitmap`을 채택한 것은 우연이 아니다.
자료구조 하나가 시스템 전체 성능의 병목을 제거할 수 있다는 사실을 보여주는 좋은 사례다.
